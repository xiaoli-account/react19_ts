# 装饰器顺序问题详解

## 问题描述

当使用多个装饰器时，如果顺序不正确，会导致**权限检查失效，接口仍然被调用**的问题。

## 错误示例

```typescript
// ❌ 错误：权限检查失效
@LeeLoggerMethod({
  level: LOG_LEVEL.INFO,
  module: "用户认证",
  operation: OPERATION_TYPE.LOGIN,
})
@LeeApiPermission("api:/login")
login(data: LoginInfo) {
  return $post("/login", data);
}
```

**现象：**
- 控制台显示权限检查失败的警告
- 但接口仍然被调用（Network 中能看到请求）
- 日志记录了"调用方法"和"接口请求失败"

## 正确示例

```typescript
// ✅ 正确：权限检查生效
@LeeApiPermission("api:/login")
@LeeLoggerMethod({
  level: LOG_LEVEL.INFO,
  module: "用户认证",
  operation: OPERATION_TYPE.LOGIN,
})
login(data: LoginInfo) {
  return $post("/login", data);
}
```

**现象：**
- 控制台显示权限检查失败的警告
- 接口**不会**被调用（Network 中没有请求）
- 没有日志记录（因为方法根本没执行）

## 原理解释

### 装饰器执行顺序

TypeScript/JavaScript 中，装饰器的执行顺序是：

1. **评估顺序**：从上到下
2. **应用顺序**：从下到上

```typescript
@A  // 第3步：最后应用（最外层）
@B  // 第2步：第二个应用
@C  // 第1步：第一个应用（最内层）
method() {}
```

等价于：

```typescript
method = A(B(C(method)))
```

### 错误顺序的执行流程

```typescript
@LeeLoggerMethod({...})  // 外层
@LeeApiPermission("api:/login")  // 内层
login(data: LoginInfo) {
  return $post("/login", data);
}
```

**执行流程：**

1. `@LeeApiPermission` 先包装 `login` 方法：
   ```typescript
   wrappedLogin1 = function() {
     if (!hasPermission) throw error;
     return originalLogin();
   }
   ```

2. `@LeeLoggerMethod` 再包装 `wrappedLogin1`：
   ```typescript
   wrappedLogin2 = function() {
     console.log("调用方法");  // ← 先执行这里
     try {
       return wrappedLogin1();  // ← 然后才检查权限
     } catch (error) {
       console.log("方法执行失败");
     }
   }
   ```

3. 调用 `login()` 时：
   - ✅ 记录"调用方法"日志
   - ✅ 调用 `wrappedLogin1()`
   - ❌ 权限检查失败，抛出错误
   - ✅ 记录"方法执行失败"日志
   - **问题：`@LeeLoggerMethod` 在第951行已经调用了 `originalMethod.call(this, ...args)`**

### 正确顺序的执行流程

```typescript
@LeeApiPermission("api:/login")  // 外层
@LeeLoggerMethod({...})  // 内层
login(data: LoginInfo) {
  return $post("/login", data);
}
```

**执行流程：**

1. `@LeeLoggerMethod` 先包装 `login` 方法：
   ```typescript
   wrappedLogin1 = function() {
     console.log("调用方法");
     return originalLogin();
   }
   ```

2. `@LeeApiPermission` 再包装 `wrappedLogin1`：
   ```typescript
   wrappedLogin2 = function() {
     if (!hasPermission) throw error;  // ← 先检查权限
     return wrappedLogin1();  // ← 有权限才执行
   }
   ```

3. 调用 `login()` 时：
   - ✅ 检查权限
   - ❌ 权限不足，抛出错误
   - ❌ **不会**执行 `wrappedLogin1()`
   - ❌ **不会**记录日志
   - ❌ **不会**调用接口

## 代码对比

### 错误顺序的实际调用链

```
用户调用 login()
  ↓
@LeeLoggerMethod 拦截
  ↓
记录"调用方法"日志 ✅
  ↓
调用 originalMethod (实际是 @LeeApiPermission 包装的方法)
  ↓
@LeeApiPermission 检查权限
  ↓
权限不足，抛出错误 ❌
  ↓
@LeeLoggerMethod 捕获错误
  ↓
记录"方法执行失败"日志 ✅
  ↓
向上抛出错误

问题：接口已经在 @LeeLoggerMethod 中被调用了！
```

### 正确顺序的实际调用链

```
用户调用 login()
  ↓
@LeeApiPermission 拦截
  ↓
检查权限
  ↓
权限不足，抛出错误 ❌
  ↓
向上抛出错误

结果：方法根本没执行，接口没被调用 ✅
```

## 通用规则

### 装饰器优先级（从高到低）

1. **权限检查** - `@LeeApiPermission` （最外层，最先执行）
2. **日志记录** - `@LeeLoggerMethod`
3. **其他装饰器**
4. **原始方法**

### 记忆口诀

**"权限第一，日志第二，其他随意"**

```typescript
@LeeApiPermission("...")     // 1️⃣ 权限检查（最重要）
@LeeLoggerMethod({...})      // 2️⃣ 日志记录
@OtherDecorator({...})       // 3️⃣ 其他装饰器
method() {                   // 4️⃣ 原始方法
  // ...
}
```

## 常见错误

### 错误 1：权限装饰器在下面

```typescript
// ❌ 错误
@LeeLoggerMethod({...})
@LeeApiPermission("api:/login")
```

**后果：** 接口仍会被调用

### 错误 2：忘记添加权限装饰器

```typescript
// ❌ 错误
@LeeLoggerMethod({...})
login(data: LoginInfo) {
  return $post("/login", data);
}
```

**后果：** 没有权限检查

### 错误 3：装饰器顺序混乱

```typescript
// ❌ 错误
@OtherDecorator({...})
@LeeApiPermission("api:/login")
@LeeLoggerMethod({...})
```

**后果：** 可能导致权限检查失效

## 最佳实践

### 1. 统一顺序

在整个项目中保持一致的装饰器顺序：

```typescript
@LeeApiPermission("...")  // 总是第一个
@LeeLoggerMethod({...})   // 总是第二个
@OtherDecorator({...})    // 其他装饰器
method() {}
```

### 2. 代码审查

在 Code Review 时，重点检查装饰器顺序。

### 3. 添加注释

```typescript
// 注意：@LeeApiPermission 必须在最上面
@LeeApiPermission("api:/login")
@LeeLoggerMethod({...})
login(data: LoginInfo) {
  return $post("/login", data);
}
```

### 4. ESLint 规则（可选）

可以配置 ESLint 规则来检查装饰器顺序。

## 总结

| 装饰器顺序 | 权限检查 | 接口调用 | 日志记录 | 正确性 |
|-----------|---------|---------|---------|--------|
| `@LeeApiPermission` → `@LeeLoggerMethod` | ✅ 生效 | ❌ 不调用 | ❌ 不记录 | ✅ 正确 |
| `@LeeLoggerMethod` → `@LeeApiPermission` | ⚠️ 检查但无效 | ❌ 仍调用 | ✅ 记录 | ❌ 错误 |

**关键点：**
- ✅ `@LeeApiPermission` **必须**在最上面
- ✅ 装饰器顺序决定了执行顺序
- ✅ 权限检查应该是第一道防线
- ✅ 无权限时，方法不应该被执行

**记住：权限装饰器永远在最上面！** 🔒
