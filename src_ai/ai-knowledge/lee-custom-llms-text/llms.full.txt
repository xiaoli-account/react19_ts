## 规则冲突处理原则

当多条规则产生冲突时，按以下优先级处理：

1. **强制规则 (FR)** > 执行规则 (ER) > **用户明确指定** > 建议规则 (AR) > 其他规则

**说明：**
- 强制规则 (FR) 拥有最高优先级，任何情况下都不可违反
- 当用户明确指定与 FR/ER 冲突时，AI 应提醒用户规则限制，无法执行用户指定
- 当用户明确指定与 AR 冲突时，遵循用户指定
- 同级规则内部：标记 ⭐⭐⭐ 的规则优先级更高

## 强制规则 (Forced Rules)
> 以下规则为强制规则，必须严格遵守。各子规则之间为"且"的关系，需全部满足。
> 标记说明：⭐⭐⭐ = 核心规则，必须优先执行
> 强制规则为最高优先级规则，其他规则与强制规则冲突时，以强制规则为准。

### FR-01 自检机制 ⭐⭐⭐
**代码生成前的自检清单（必须执行）：**
  ✓ 这是独立功能/页面/模块吗？
  ✓ 预估代码总行数是多少？
  ✓ 是否会影响项目运行？
  ✓ 应该放在 src/ 还是 src_ai/？
  ✓ 是否符合技术栈约束？
  
**判断逻辑：**
  - 独立功能 OR 代码 ≥ 500 行 → `src_ai/`
  - 非独立功能 AND 代码 < 500 行 → `src/`

### FR-02 代码安全性
AI 生成的代码不能影响项目运行，不能破坏项目结构，不能破坏项目功能。

### FR-03 代码行数预估
生成代码前需先预估本次沟通需要生成的所有文件的总代码行数。

### FR-04 目录结构理解
当前项目的 `src/` 目录与 `src_ai/` 目录是并列的：
  * `src/` - 手动编写的项目代码
  * `src_ai/` - AI 生成的代码

### FR-05 目录作用定位
`src_ai/` 目录是正式项目不可分割的一部分，与 `src/` 目录共同构成完整的项目。
  * a. `src/` 为项目主目录：手动编写的项目代码
  * b. `src_ai/` 为辅助目录：AI 生成的代码，为 src 目录提供辅助

### FR-06 代码存放规则 ⭐⭐⭐
**按优先级判断（从高到低）：**
  * **a. 独立功能规则**：AI 代码以任何方式实现一个独立的功能、页面、模块、单元、组件及其他高内聚低耦合的代码时，无需与发起人进行确认（AI在src_ai目录拥有所有决策权），直接放入 `src_ai/` 目录下，严禁放入 `src/` 目录下。
    - ⚠️ 例外：用户明确指定存放路径时，提醒用户当前项目规则限制，无法由 AI 决定本次存放路径
  * **b. 大文件规则**：当 AI 代码超过 500 行时，必须将代码放置在 `src_ai/` 目录下。
  * **c. 小文件规则**：当 AI 代码小于 500 行且非独立功能时，可以将代码放置在 `src/` 目录下，保持发起者本意。

### FR-07 src_ai 目录作用
`src_ai/` 目录下的代码主要作用是：
  * 区分项目内 AI 生成的代码与手动编写的代码，便于管理和维护
  * 不是作为中间件为 src 目录提供审查、测试等功能测试通过后迁移至 src 目录使用

### FR-08 测试代码存放
测试相关代码，存放至 `src_ai/test/` 目录下。

### FR-09 文档管理规则
**文档命名规则：**
  - 文档命名默认使用中文
  - 用户指定文档命名时，以用户指定为准

**文档存放规则：**
  - AI 主动生成的文档 → `docs/ai_docs/{文档作用类型}/`
  - 用户明确指定路径的文档 → 按用户指定路径存放
  
**文档作用类型分类：**
  
  **1. 使用文档类**
  - `guide/` - 使用指南、快速开始、操作手册、最佳实践
    - 示例：`dashboard-guide.md`、`quick-start.md`、`best-practices.md`

## 执行规则 (Execution Rules)
> 以下规则为执行规则，在开发过程中必须遵守。
> 执行规则与强制规则冲突时，以强制规则为准。

### ER-01 技术栈约束

- 使用 React 19.2.x + TypeScript 5.9.x + Vite 7.2.x 作为核心技术栈
- UI 组件库使用 Ant Design 6.1.x，不要使用其他 UI 框架
- 状态管理使用 Zustand 5.0.x，不要使用 Redux 或 MobX
- 路由管理使用 React Router 7.12.x
- 国际化使用 i18next 25.7.x + react-i18next 16.5.x
- HTTP 请求使用 Axios 1.13.x
- 样式方案支持 SCSS 1.97.x 和 TailwindCSS 4.1.x
- 图表使用 ECharts 6.x

### ER-02 命名规范

- 目录命名：使用 kebab-case（例如：`user-management/`）
- 页面入口：统一使用 `index.tsx`
- 组件文件：统一使用 `index.tsx`
- Hooks 文件：使用 `use-xxx.ts` 格式
- 样式文件：使用 `styles.scss` 或 `styles.module.scss`
- SCSS 变量：业务层使用 `$lee-xxx`，框架层使用 `$lee-layout-xxx`
- CSS 变量：业务层使用 `--lee-xxx`，框架层使用 `--lee-basic-xxx` / `--lee-sidebar-xxx`
- 国际化命名空间：框架层使用 `lee-layout-` 前缀，业务层使用页面/模块名

### ER-03 项目结构约定
> 以下约定适用于 `src/` 目录下的手动编写代码。
> AI 生成的代码应遵循强制规则，满足强制规则后放入 `src_ai/` 目录。

- 全局通用组件放在 `src/components/`
- 路由级页面放在 `src/pages/`
- 页面私有组件放在对应页面目录下的 `components/` 子目录
- 通用 Hooks 放在 `src/hooks/`
- 页面私有 Hooks 放在对应页面目录下
- API 服务层放在 `src/services/`
- 全局状态管理放在 `src/store/`
- 工具函数放在 `src/utils/`

### AR-06 开发示例

#### 创建新页面

1. 在 `src/pages/` 下创建页面目录（kebab-case）
2. 创建 `index.tsx` 作为页面入口
3. 如需样式，创建 `styles.scss`
4. 如需类型定义，创建 `types.ts`
5. 在 `src/router/routes.ts` 中添加路由配置
6. 在国际化文件中添加对应的翻译

#### 创建全局组件

1. 在 `src/components/` 下创建组件目录（kebab-case）
2. 创建 `index.tsx` 作为组件入口
3. 如需样式，创建 `styles.scss` 或 `styles.module.scss`
4. 如需类型定义，创建 `types.ts`
5. 导出组件供其他模块使用

#### 添加 API 服务

1. 在 `src/services/` 下创建服务文件（例如：`user.ts`）
2. 使用 `src/utils/request.ts` 中封装的 axios 实例
3. 定义请求参数和响应类型
4. 导出 API 函数供页面或 Hooks 调用

#### 创建全局状态

1. 在 `src/store/` 下创建 store 文件（例如：`user.ts`）
2. 使用 Zustand 的 `create` 函数创建 store
3. 使用 `persist` 中间件实现状态持久化（可选）
4. 导出 store hook 供组件使用

### AR-07 常见问题

#### 路径别名

- 使用 `@/` 作为 `src/` 目录的别名
- 例如：`import { request } from '@/utils/request'`

#### 样式导入

- 全局样式变量已通过 Vite 配置自动注入，无需手动导入
- 在 SCSS 文件中可直接使用 `$lee-xxx` 变量

#### 国际化使用

- 使用 `useTranslation()` Hook 获取 `t` 函数
- 调用 `t('namespace.key')` 获取翻译文本
- 框架层使用 `lee-layout-` 命名空间，业务层使用页面名作为命名空间

#### 主题切换

- 使用 `useTheme()` Hook 获取当前主题和切换函数
- 主题值为 `'light'` 或 `'dark'`
- 主题切换会自动更新 `data-theme` 属性和 CSS 变量

#### 布局切换

- 使用 `useLayoutStore()` 获取布局状态和切换函数
- 布局模式：`'basic'` / `'sidebar'` / `'top-menu'`
- 布局配置会自动持久化到 localStorage
